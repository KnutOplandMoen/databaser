Interaktiv reise

Spørringens reise gjennom DBMS-en

Følg en SELECT fra klient til disk og tilbake. Hvert trinn knytter sammen ulike kapitler i kurset.

  1. 01 Klient sender SQL
  2. 02 Parser bygger AST
  3. 03 Katalog og typesjekk
  4. 04 Optimizer velger plan
  5. 05 Lock og log engasjeres
  6. 06 Buffer-oppslag
  7. 07 Disk-I/O
  8. 08 B+-tree-traversering
  9. 09 Resultat og commit
Trinn 01

Klient sender SQL

[Stubber:]

  • Klient åpner forbindelse (TCP eller socket)
  • Autentisering, session-state
  • SQL-tekst sendes som bytes

Relevant: SQL-grunnform, autorisasjon.

Trinn 02

Parser bygger AST

  • Tokenizer → token-strøm
  • Parser → abstrakt syntakstre (AST)
  • Syntaksfeil rapporteres her
Trinn 03

Katalog og typesjekk

  • SQL-katalogen slår opp tabell- og kolonneinformasjon
  • Typesjekk og navneoppløsning
  • Privilegiekontroll
  • AST → algebra-tre
Trinn 04

Optimizer velger plan

  • Algebraiske ekvivalenser brukes (push down σ, omorganiser join-rekkefølge)
  • Statistikk fra katalogen + selektivitet
  • Velger fysisk plan: indeks-scan, hash join, …

Relevant: relasjonsalgebra, kostnadsestimering.

Trinn 05

Lock og log engasjeres

  • Executor begynner å kjøre planen
  • Lock manager (eller MVCC) gir tilgang til data
  • Log writer skriver begin-record

Relevant: samtidighet, recovery.

Trinn 06

Buffer-oppslag

  • Trenger blokk N — er den i buffer?
  • Hash-indeks på BlockId
  • Hit: returner peker. Miss: gå til disk.

Relevant: database-buffer.

Trinn 07

Disk-I/O

  • Eviction av en eldre frame (kanskje skitten — flush først)
  • Les blokk fra disk via deviceId + offset
  • Verifiser flip/flop-stempel
Trinn 08

B+-tree-traversering

  • Hvis indeks-scan: gå fra rot til blad
  • Hver node er én blokk → typisk 3–4 nivåer for 100M rader
  • Range scan: følg blad-link

Relevant: B+-trær.

Trinn 09

Resultat og commit

  • Tupler streames tilbake til klient
  • Ved skriv: commit-record skrives, log-buffer flushes (WAL)
  • Låser slippes (etter strict 2PL)

Relevant: WAL.
