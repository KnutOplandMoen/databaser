01 · Hvorfor

Klassiske anomalier

[Stubber:]

  • Lost update

02 · 2PL

To-fase låsing

  • Growing phase — bare ta låser
  • Shrinking phase — bare slipp låser
  • Garanterer conflict-serializability
  • Shared (S) vs. exclusive (X) låser
  • Lock compatibility-matrise
03 · Strict og rigorous 2PL

Hold låsene til commit

  • Strict 2PL — hold X-låser til commit/abort
  • Rigorous 2PL — hold alle låser til commit/abort
  • Begge gir cascadeless schedules
04 · Deadlock

To transaksjoner venter på hverandre

  • Wait-for-graf — sykel ⇒ deadlock
  • Deteksjon: sjekk graf med jevne mellomrom, abort ofre
  • Forebygging: wait-die, wound-wait
  • Timeout-basert
05 · Lock granularity

Hva låser vi?

  • Hierarki: database → tabell → side → record
  • Intent locks (IS, IX, SIX)
  • Trade-off: granular = mindre konflikter, mer overhead
06 · Timestamp ordering

Bytt låser med tidsstempler

  • Hver transaksjon får TS ved start
  • Read/write tester TS-orden, abort hvis brutt
  • Thomas write rule
  • Optimistic concurrency control (OCC)
07 · MVCC

Multi-version concurrency control

  • Hver write lager ny versjon
  • Reads ser konsistent snapshot — sliper ikke å låse
  • Snapshot isolation
  • Brukt i PostgreSQL, Oracle, mange andre
08 · Isolation levels

SQL-standardens fire nivåer

  • READ UNCOMMITTED — dirty reads tillatt
  • READ COMMITTED — hindrer dirty reads
  • REPEATABLE READ — hindrer non-repeatable
  • SERIALIZABLE — som seriell
  • Hva snapshot isolation faktisk gir (write skew-anomali)
